## Хранилище данных

На текущем этапе логичнее начать с JSON-хранилища внутри `YClients-Dashboard`, потому что:

- текущий dashboard уже использует файловые данные;
- админка пока только для одного пользователя;
- не нужны новые зависимости;
- быстрее запустить рабочий прототип;
- проще контролировать публичный и приватный срезы данных.

Рекомендуемая эволюция:

1. Сейчас: JSON-файлы в `YClients-Dashboard/data/mixology/`.
2. Следующий этап: SQLite, когда появится регулярное редактирование, история изменений и больше сущностей.
3. PostgreSQL: только при многопользовательской админке, ролях, сложных выборках и высокой нагрузке.

Важно: даже на JSON-этапе нужно делать сервисный слой, чтобы потом заменить хранилище без переписывания публичного API и админки.

## API-вектор

```text

```

```text
/api/public/mixology/...
```

Публичный endpoint должен отдавать только безопасный срез данных.

Пример будущего публичного endpoint:


Он может отдавать:

Он не должен отдавать:

- телефоны;
- email;

- токены;
- приватные комментарии;
- ссылки на Google Drive;
- служебные поля Google Sheets.

Принципы:

- все чтение и запись данных проходят через сервис;
- запись делается аккуратно, с backup;
- публичный serializer отдельно чистит приватные данные;
- frontend dashboard не читает приватные файлы напрямую.

### Этап 4. Публичное API

Добавить endpoint:

```text
/api/public/mixology/page
```

Проверить:

- отсутствие приватных полей;
- стабильный формат ответа;
- кеширование;
- понятные ошибки.

### Этап 5. Админка редактирования

Постепенно добавить управление:

- сезонами;

- победителями;
- судейскими итогами;
- партнерами;
- фотоальбомами;
- публичными текстами карточек;
- статусом публикации.

### Этап 6. Интеграции

Подключать по одной:

- yClients;
- Google Sheets;
- фотоархив чемпионатов;
- Битрикс, если понадобится.

Каждая интеграция сначала пишет во внутренний слой данных, а наружу отдается только публичный срез.

### Этап 7. Переход публичной страницы на backend

Публичная Tilda-страница пока остается основной.

Дальше возможны два варианта:

1. Генератор остается основным, но берет данные из backend/export JSON.
2. Tilda-страница напрямую запрашивает публичный API.

Решение нужно принимать после появления стабильного backend API.

## UX/UI-ориентиры

- Страница должна быть полезна участникам, зрителям, партнерам и будущим участникам.
- Фильтры должны быть понятными без внутреннего контекста.
- Карточки победителей должны открываться по клику.
- В карточке победителя должны быть фото, напиток, презентация, место, баллы и судьи.
- Фотоальбомы должны открываться в popup-просмотре, а не вести на Google Drive.
- Зеленые кнопки должны иметь белый текст.
- Дизайн сверять с `DESIGN_SYSTEM.md`.
- Не перегружать страницу техническим текстом.

## Риски

### Утечка приватных данных

Главный риск - случайно вывести приватные поля в публичный HTML или публичный API.

Защита:

- отдельный public serializer;
- blacklist приватных полей;
- проверки перед публикацией;
- отсутствие Google Drive ссылок в публичной выдаче.

### Большой server.js

В `YClients-Dashboard` уже большой `server.js`. Нельзя складывать туда всю новую бизнес-логику.

Защита:

- в `server.js` только routes;
- логика в `src/mixology-service.js`;
- публичная очистка в `src/mixology-public-serializer.js`.

### JSON-хранилище

JSON подходит для первого этапа, но плохо масштабируется при многопользовательском редактировании.

Защита:

- atomic write;
- backup перед записью;
- подготовить структуру так, чтобы позже перейти на SQLite.

### CORS и авторизация

Tilda должна получать только публичные endpoints. Закрытая админка должна оставаться за авторизацией dashboard.

Защита:

- публичные endpoints только под `/api/public/mixology/...`;
- admin endpoints только под закрытой авторизацией;
- не смешивать public и admin data.
